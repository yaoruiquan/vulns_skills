## 2. `form_context.json` 字段

浏览器填表阶段只读取 `/tmp/vulns-skills/phase2-ncc-report/form-contexts/YYYY-MM/DAS-ID/form_context.json`。`extract_vuln_data.py` 是底层提取脚本，不直接作为浏览器阶段的数据源；运行时 JSON 不写入正式提交材料目录。

| 字段 | docx 来源 | 说明 |
|----------|-----------|------|
| `title` | `漏洞名称` | 对应页面“漏洞名称” |
| `description` | `漏洞描述`、`漏洞简介` | 对应页面“漏洞描述” |
| `impact` | `漏洞危害`、`漏洞分析` | 对应页面“漏洞危害” |
| `vuln_type` | `漏洞类型` | 仅作为类型判断依据，页面实际选项需人工确认一次 |
| `target_type` | `影响对象类型` | 对应页面“影响对象” |
| `unit_name` | `漏洞厂商`、`影响厂商`、`厂商名称`、`提交机构` | 对应页面“漏洞厂商” |
| `affected_product` | `影响产品`、`受影响实体`、`产品名称` | 对应页面“影响组件” |
| `version` | `影响版本`、`受影响实体版本号`、`版本号` | 对应页面“影响版本” |
| `url` | `漏洞URL`、`漏洞定位` | 对应页面“漏洞 URL” |
| `verification` | `漏洞验证过程`、`验证过程`、`漏洞验证` | 可作为补充说明 |
| `temporary_solution` | `临时解决方案` | 对应“修复方案说明”候选内容 |
| `formal_solution` | `正式解决方案`、`修复方案` | 对应“修复方案说明”候选内容 |
| `submit_date` | `提交日期` | 仅供参考，不直接当作“发现日期” |
| `docx_path` | 材料目录 | 当前使用的源 `docx` |
| `upload_zip_path` | `poc/*.zip` | 第一版优先上传的 zip |
| `screenshot_paths` | `poc验证图片/*` | 可选补充附件 |
| `video_paths` | `poc验证视频/*` | 可选补充附件 |

## 3. 当前填写规则

| 平台字段 | 规则 |
|----------|------|
| 是否为原创漏洞 | 固定选择“是” |
| 发现日期 | 以页面允许的日期控件或当前材料确认，不直接使用 `提交日期` 硬填 |
| 漏洞类型 | 先按页面一级分类选择，不能只把 docx 的 `漏洞类型` 原样塞进去 |
| 影响对象 | 优先根据 `target_type` 选择 |
| 漏洞厂商 | 使用 `unit_name` |
| 影响组件 | 使用 `affected_product` |
| 影响版本 | 使用 `version` |
| 漏洞名称 | 使用 `title` |
| 漏洞详细分类 | 需要根据页面下拉值和材料内容人工确认一次 |
| 漏洞 URL | 使用 `url`；如果材料里就是 `http://none`，保持原样，不要自行改写 |
| 漏洞描述 | 使用 `description` |
| 漏洞危害 | 优先使用 `impact`；如果材料没有单独危害字段，可人工摘取关键危害结论 |
| 修复方案 | 根据页面单选按钮选择，再把 `temporary_solution` 或 `formal_solution` 写入“修复方案说明” |
| 漏洞附件 | 第一版优先上传 `upload_zip_path`；如页面支持多文件，再补充 `docx/截图/视频` |

## 4. 不要硬填的内容

- 不要把 `提交日期` 当成 `发现日期` 自动写入。
- 不要把 `漏洞类型=二进制` 直接当作页面下拉值。
- 不要凭空生成“漏洞详细分类”。
- 不要自行编造厂商官网、修复方案或验证过程。

## 5. 首次联调后需要补充

NCC 平台实际 `uid`、下拉可选值、提交成功页元素仍需在真实页面里用 MCP `take_snapshot` 确认，随后补充到：

- `references/selectors.md`
- 本文件的“当前填写规则”
